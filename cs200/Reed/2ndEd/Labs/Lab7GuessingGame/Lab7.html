Binary reduction guessing game

For this lab, you are to design and implement a page for playing a guessing game with the user. The game begins by asking the user to think of a number in some range, such as 1 to 100. Your program should then repeatedly guess numbers until it gets the user's number. After each guess, the program will receive feedback from the user as to whether the guess was correct, too high, or too low. The number of guesses required to get the user's number should be displayed at the end. Your page should appear as follows, with text boxes that allow the user to specify the range of numbers and a button for starting the game:

When the user clicks on the button, they should be shown the program's initial guess, and prompted to see whether that guess is correct, too high, or too low. The program should continue generating guesses and prompting for responses until it finally guesses the user's number.

One way to think of the game from the program's perspective is to focus on the range in which the user's number must lie. At the beginning of the game, the user's number could be anywhere in the initial range, say 1 to 100. By making a guess, such as 50, the computer is able to reduce the range in which the user's number can lie. For example, if 50 was too low, then the number must lie in the range 51 to 100. Likewise, if 50 was too high, then the number must lie in the range 1 to 49. Subsequent guesses can reduce the range even further, repeating the process until the program eventually hones in on and guesses the number. Since the program should try to guess the user's number in the fewest number of guesses, it should always guess the midpoint of the possible range of numbers in order to divide the possible range in half. Because of the halving effect, this strategy is known as binary reduction..

Iterative refinement

Before you begin designing and implementing a program for playing the binary reduction guessing game, it is worthwhile to take a moment and think about program design. Designing a program from scratch is not always a simple task. Without forethought and planning, it is easy to become overwhelmed in complexity and lose focus when writing code. In fact, Henry Ledgard once remarked:

The sooner you start coding your program, the longer it's going to take.

A common, systematic approach to designing programs is known as iterative refinement. The basic idea is to start with a general description of a solution to the given problem, and then incrementally add more and more detail until the final program is derived. The main advantage of an incremental approach is that it helps to manage complexity. At the start, you can abstract away some of the details in order to describe a general outline to the solution. Then, you can add details slowly, a little bit at a time, so that no single step in the design process is overwhelmingly complex. In this way, a series of small refinements to a general outline can lead to the solution of a complex problem.
