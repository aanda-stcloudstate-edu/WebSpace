B+ Tree Group Project

This project is the final phase of a four-phase scalable sequence-set with index project resulting in a secondary storage-based, B+tree.

Objective

Generate a B+tree as a file composed of linked, fixed-size blocks.
Each of these fixed-size blocks will either be an index set (B-tree) block or [doubly-linked] sequence-set block (Group Project 3.0).


Develop and test a static version of the B+tree Class which can generate a B+tree file,
then verify that the file can be used as a sequence set and as an index to access any record through the root via the record's primary key.
Extra credit:Then, add dynamic functionality to be able to insert and delete records.

You must design your B+tree using the specifications in Folk Section 9.10, with the sequence set of pages/blocks (containing records) being doubly linked, as you produced in the final Sequence Set project (Group Project 3.0).
The B-Tree pages/blocks, growing up from the sequence set leaves, will contain only lists of ordered pairs of the largest keys of each child page/block with the relative block number (RBN) of that child, sorted by key.
The B-tree index (growing up from the sequence set leaves) will replace the simple index file (from the previous project), with the crucial factors being that the B-tree index will use the same page/block size as the sequence set, and the B-tree index pages/blocks will be interleaved with the sequence set pages/blocks in the same file.

Teams will each implement a static B+tree Class that generates and uses a paged/blocked B+tree file.
Only a minimum set of block and record objects will be kept in RAM at any one time.
Your team must implement your B+tree so that your B+tree only operates from the file in which it is saved.
The B+tree must not reside in RAM.
The only B+tree component which should continually reside in RAM while the B+tree file is open will be its header record.
(it would also be OK to keep the index root block in RAM, as all searches pass through the index root block)

The same US Postal Code data file will be used (from the previous project),
however, you should test the flexibility of your design by successfully processing a column-reordered version of the data file, and one with fewer fields.

Create a method of printing (dumping) the contents and shape of your B-tree file so that you can determine whether your implementations for building, insertion and deletion are functioning properly. This dump should be sufficiently readable to help facilitate debugging. Write a test driver that demonstrates that your B+tree Class and file work properly and robustly.

Development tips

  • Even though the sequence-set and index-set blocks are the same size, each block type has a different architecture. Therefore, I recommend using a differnt buffer type for each to allow and facilitate each block type having its own specialized pack and unpack methods.
  • If the primary key and the RBN fields are fixed length, then the index set blocks can have a fixed well-defined order.
