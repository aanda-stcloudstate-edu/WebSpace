Sequence Set Group Project

Preliminary Documents Drafts

Prior to commencing coding for the Sequence Set Class assignment, your group will submit a set of preliminary documents which specify clearly the components of the Sequence Set project and who is responsible for which parts.

WHAT TO TURN IN:

  • Design Document for the Sequence Set Class.
    (Consider using facilities in, e.g. Jgrasp or Doxygen)
    • Describe the architecture of:
      (both as objects in memory and as corresponding entities in a file)
      {For entities in files, include and describe all of the non-data overhead}
      • the input data file
      • the blocked record file
      • the index file (maps primary keys to RBNs)
      • the header record
      • a block containing records
      • a record (& its fields)
    • Describe the methods of the SS class:
      • constructor(s) / destructor (etc. for in-RAM objects)
      • create
      • open/load (necessary components of an existing SS {i.e. header record & index file into memory)
        • consider optionally running the validate method
      • close
      • is_empty (via a flag, can be applied to either file, a block, a record slot in a block, or a field within a record)
      • search (for a record)
      • populate (populate the blocked record file from the input data file)
        • consider populating to 3/4 capacity as a default parameter
          (can be changed for testing block merging, splitting, & record redistribution)
      • insert (a record)
      • delete (a record)
      • update (a field of a record)
      • display_record
      • diplay_SS (parameterized to display the whole record or a subset of fields)
      • validate (is your sequence set ordered by primary key? Can you get to each record via the index file?)
      • (...private helper functions/methods)
      • (...debug functions/methods) {consider using a static debug flag for the class}
  • Test Document
    • Describe the test driver program
      • describe how to use the command line options
    • Describe the details and purpose of each running of the test driver program in the test script
      • describe the method/functionality that will be tested
        • For verifying correct performance of record deletion or insertion, include tests which demonstrate (each of) these effects:
          1. no structural change, & the operation is confined to that block
            • this may require an update to the index
            • (this will be the norm for larger capacity blocks filled initially to 75%)
          2. no structural change, but there is redistribution operation performed with one of the two logically adjacent blocks
            • this will require an update to the index
          3. a structural change occurs — either a merge or a split
            • this will require an update to the index
          4. how do you handle a failed deletion search?
        • (suggestion:) an insertion or a deletion test can have the following sequential structure:
          1. populate
          2. display structure and/or run validate
          3. perform insertion or deletion
          4. display structure and/or run validate
      • Suggested command line structure:
        • must have the filename for that subset of the full data set which will be used for initially populating the sequence set file for this test
        • can additionally have a list of one or more primary keys, each prefaced with either a delete flag or an insert flag, which will be performed, in sequence, after the SS is populated. For insertion, the primary key will be used to provide the full record for that key.

    • Describe the set of tests in the test script.
